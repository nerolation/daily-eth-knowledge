The Shuffling Algorithm

compute_shuffled_index, swap-or-not, and why 90 rounds

📍 Spec Location

phase0/beacon-chain.md → compute_shuffled_index

💡 Core Concept

The shuffling algorithm is how Ethereum consensus assigns validators to committees unpredictably but deterministically. Every validator needs to know their duties, but no one should be able to predict assignments far in advance (enables attacks).

The algorithm is called swap-or-not — a cryptographic construction that shuffles by iteratively deciding whether each index should swap with another position.

Key insight: Rather than shuffling an entire array (expensive), we compute the shuffled position of a single index on-demand. This is crucial for light clients and efficient implementations.

🛠 The Code

SHUFFLE_ROUND_COUNT = 90  # Why 90? We'll get there.

def compute_shuffled_index(
    index: uint64, 
    index_count: uint64, 
    seed: Bytes32
) -> uint64:
    """
    Return the shuffled index corresponding to ``seed`` 
    (and ``index_count``).
    """
    assert index < index_count

    # Swap or not (https://link.springer.com/content/pdf/10.1007%2F978-3-642-32009-5_1.pdf)
    # See the 'generalized domain' algorithm on page 3
    for current_round in range(SHUFFLE_ROUND_COUNT):
        # 1. Compute pivot for this round
        pivot = bytes_to_uint64(
            hash(seed + uint_to_bytes(uint8(current_round)))[0:8]
        ) % index_count
        
        # 2. Compute the "flip" position
        flip = (pivot + index_count - index) % index_count
        
        # 3. Get deterministic bit to decide swap
        position = max(index, flip)
        source = hash(
            seed + 
            uint_to_bytes(uint8(current_round)) + 
            uint_to_bytes(uint32(position // 256))
        )
        byte = uint8(source[(position % 256) // 8])
        bit = (byte >> (position % 8)) % 2
        
        # 4. Swap or not
        index = flip if bit else index

    return index

🔬 Step-by-Step Breakdown

Each round does this:

1. Pivot calculation

pivot = hash(seed || round)[0:8] % index_count

The pivot is a random position in [0, index_count) derived from the seed and round number.

2. Flip calculation

flip = (pivot + index_count - index) % index_count

This computes the "mirror" position of index around the pivot. If index = pivot + k, then flip = pivot - k (mod index_count).

3. Bit extraction

position = max(index, flip)  # Ensures both sides compute same bit
source = hash(seed || round || position // 256)
bit = (source[position % 256 // 8] >> (position % 8)) % 2

We need a deterministic bit to decide the swap. Using max(index, flip) ensures both positions looking at each other get the same bit (critical for consistency).

4. Conditional swap

index = flip if bit else index

If bit=1, swap positions. If bit=0, stay put.

❓ Why 90 Rounds?

The swap-or-not algorithm achieves perfect uniformity after enough rounds. The original paper proves:

  • For n elements, O(log n) rounds suffice for statistical closeness to uniform
  • 90 rounds provides ~2-40 statistical distance for any realistic validator count

With ~1M validators, log₂(1M) ≈ 20. Why 90 then? Security margin. The spec uses 90 to ensure the shuffle is indistinguishable from random even under adversarial analysis.

SHUFFLE_ROUND_COUNT = 90  # Provides ~40-bit security margin

🎨 Why This Design?

Alternative 1: Fisher-Yates shuffle

  • Shuffles entire array: O(n) time, O(n) space
  • Problem: Can't compute single index without full shuffle
  • Light clients would need to process all validators

Alternative 2: Hash-based random assignment

def naive_assignment(index, seed, committee_count):
    return hash(seed || index) % committee_count
  • Problem: Uneven distribution (birthday paradox creates variance)
  • Some committees could be 50% larger/smaller than target

Swap-or-not advantages:

  1. Single-index computation: O(90) hashes per index
  2. Perfect uniformity: Provably uniform permutation
  3. Reversibility: Run backwards with same seed to unshuffle
  4. Deterministic: Same seed → same permutation everywhere

🔗 Where Shuffling Gets Used

def compute_committee(indices, seed, index, count):
    start = (len(indices) * index) // count
    end = (len(indices) * (index + 1)) // count
    return [
        indices[compute_shuffled_index(i, len(indices), seed)]
        for i in range(start, end)
    ]

The shuffle distributes validators across committees. Each committee gets a contiguous range of the shuffled indices.

The seed comes from:

def get_seed(state, epoch, domain_type):
    mix = get_randao_mix(
        state, 
        epoch + EPOCHS_PER_HISTORICAL_VECTOR - MIN_SEED_LOOKAHEAD - 1
    )
    return hash(domain_type + uint_to_bytes(epoch) + mix)

RANDAO mix from a past epoch → deterministic seed → deterministic shuffle.

⚠️ Gotchas & Edge Cases

1. position = max(index, flip)

This is subtle but critical. Both index=A and index=B (where B is A's flip) must see the same bit. Using max() ensures they hash the same position.

2. Hash caching matters

Each round computes:

  • 1 hash for pivot
  • 1 hash for the bit source (per 256 positions)

Naive implementation: 90 × 2 = 180 hashes per index.
Optimized: Cache pivot hashes, batch position hashes → ~90-100 hashes.

3. Empty committee edge case

assert index < index_count  # Fails if index_count = 0

The spec handles this upstream—committees always have at least 1 validator.
