💡 The Core Problem

Ethereum has ~1M validators. Each slot, ~12,500 validators attest (1/32 of the set). If we stored each attestation individually:

  • 1 signature per validator = 96 bytes × 12,500 = 1.2 MB per slot just for signatures

BLS signatures solve this: aggregate N signatures into one 96-byte signature. But we need a data structure to track which validators participated.

🛠️ Two Attestation Formats

1. Attestation (network/block format)

class Attestation(Container):
    aggregation_bits: Bitlist[MAX_VALIDATORS_PER_COMMITTEE]
    data: AttestationData
    signature: BLSSignature  # Aggregated!

The aggregation_bits is a bitfield: position i is set if committee member i participated.

2. IndexedAttestation (validation format)

class IndexedAttestation(Container):
    attesting_indices: List[ValidatorIndex, MAX_VALIDATORS_PER_COMMITTEE]
    data: AttestationData
    signature: BLSSignature

attesting_indices lists the actual validator indices — needed to look up public keys for signature verification.

🔄 The Conversion: get_indexed_attestation

def get_indexed_attestation(state: BeaconState, attestation: Attestation) -> IndexedAttestation:
    attesting_indices = get_attesting_indices(state, attestation)
    return IndexedAttestation(
        attesting_indices=sorted(attesting_indices),  # SORTED!
        data=attestation.data,
        signature=attestation.signature,
    )

The key helper — get_attesting_indices:

def get_attesting_indices(state: BeaconState, attestation: Attestation) -> Set[ValidatorIndex]:
    committee = get_beacon_committee(state, attestation.data.slot, attestation.data.index)
    return set(index for i, index in enumerate(committee) 
               if attestation.aggregation_bits[i])

The flow:

  1. Look up the committee for that slot/index
  2. For each set bit in aggregation_bits, include that committee member's ValidatorIndex
  3. Sort the indices for the indexed attestation

⚠️ Why Sorted Indices?

BLS aggregate signature verification requires knowing the exact order of public keys. The spec mandates:

# In is_valid_indexed_attestation:
indices = indexed_attestation.attesting_indices
if len(indices) == 0 or not indices == sorted(set(indices)):
    return False  # Must be sorted AND unique

Sorting provides:

  1. Determinism — all clients produce identical IndexedAttestation from same Attestation
  2. Uniqueness check — sorted(set(...)) catches duplicates
  3. Verification simplicity — pubkeys in predictable order

🔐 BLS Signature Aggregation

# How aggregation works (conceptual):
# Validator 1: sig_1 = Sign(sk_1, message)
# Validator 2: sig_2 = Sign(sk_2, message)
# ...
# Aggregate: agg_sig = Aggregate([sig_1, sig_2, ...])

# Verification:
def is_valid_indexed_attestation(state, indexed_attestation):
    indices = indexed_attestation.attesting_indices
    
    # 1. Sorted + unique check
    if len(indices) == 0 or not indices == sorted(set(indices)):
        return False
    
    # 2. Get all pubkeys in order
    pubkeys = [state.validators[i].pubkey for i in indices]
    
    # 3. Compute what they signed
    domain = get_domain(state, DOMAIN_BEACON_ATTESTER, 
                        indexed_attestation.data.target.epoch)
    signing_root = compute_signing_root(indexed_attestation.data, domain)
    
    # 4. Verify aggregate signature
    return bls.FastAggregateVerify(pubkeys, signing_root, 
                                   indexed_attestation.signature)

FastAggregateVerify checks that the aggregate signature is valid for:

  • The same message (signing_root)
  • All pubkeys in the list

This is O(n) in pubkeys but only one pairing for the signature — massive savings vs. n individual verifications.

⚡ Electra Changes (EIP-7549)

Pre-Electra: each committee produces separate attestations. Post-Electra: aggregate across committees.

Modified Attestation

class Attestation(Container):
    # NOW: covers ALL committees in the slot
    aggregation_bits: Bitlist[MAX_VALIDATORS_PER_COMMITTEE * MAX_COMMITTEES_PER_SLOT]
    data: AttestationData  # index field now unused!
    signature: BLSSignature
    committee_bits: Bitvector[MAX_COMMITTEES_PER_SLOT]  # NEW: which committees

Modified IndexedAttestation

class IndexedAttestation(Container):
    # Larger list — can hold validators from multiple committees
    attesting_indices: List[ValidatorIndex, 
                            MAX_VALIDATORS_PER_COMMITTEE * MAX_COMMITTEES_PER_SLOT]
    data: AttestationData
    signature: BLSSignature

Modified get_attesting_indices

def get_attesting_indices(state, attestation):
    output: Set[ValidatorIndex] = set()
    committee_indices = get_committee_indices(attestation.committee_bits)
    committee_offset = 0
    
    for committee_index in committee_indices:
        committee = get_beacon_committee(state, attestation.data.slot, committee_index)
        committee_attesters = set(
            attester_index
            for i, attester_index in enumerate(committee)
            if attestation.aggregation_bits[committee_offset + i]
        )
        output = output.union(committee_attesters)
        committee_offset += len(committee)
    
    return output

Impact: ~4x reduction in attestation data per block.

🔗 Connections

  • AttesterSlashing uses IndexedAttestation — needs explicit indices to identify who to slash
  • Fork choice uses get_attesting_indices to weight votes
  • Rewards use get_attesting_indices to credit participation
  • Aggregators (Lesson 38) select who produces aggregates

⚠️ Gotchas

  1. Empty attestation = invalid. len(indices) == 0 fails validation.
  2. Duplicates are forbidden. sorted(set(indices)) catches this — a validator can't vote twice in one attestation.
  3. Same message requirement. BLS aggregation only works when all signers signed the same AttestationData. Different data = different aggregates.
  4. Committee ≠ Indices. Committee is a position list (who's in slot 0, 1, 2...). Indices are validator IDs. The bitfield maps positions to participation.
