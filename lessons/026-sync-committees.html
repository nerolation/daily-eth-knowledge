Lesson 26: Sync Committees (Altair)

Module: Committees and Shuffling (3/8) | Date: March 1, 2026 | Spec: altair/beacon-chain.md

📚 Topic

The 512-validator committees that sign every block, enabling light clients to track the chain with minimal data.

🔍 Spec Location

altair/beacon-chain.md → SyncCommittee, get_next_sync_committee_indices, process_sync_aggregate

💡 Core Concept

Sync committees solve a fundamental problem: how can a resource-constrained device verify Ethereum's head without processing the full validator set?

Before Altair, you'd need ~500k validators' attestations to verify the head. That's impractical for phones, IoT devices, or cross-chain bridges running in blockchain VMs.

The insight: instead of having everyone attest, have a small rotating committee sign the block root each slot. Light clients only need to track this committee (512 validators) to verify blocks.

Key Parameters

Parameter Value Description
SYNC_COMMITTEE_SIZE 512 Validators per committee
EPOCHS_PER_SYNC_COMMITTEE_PERIOD 256 Epochs per rotation (~27.3 hours)

256 epochs × 32 slots × 12 seconds = ~27.3 hours per committee period.

🛠 Data Structures

SyncCommittee

The committee itself:

class SyncCommittee(Container):
    pubkeys: Vector[BLSPubkey, SYNC_COMMITTEE_SIZE]  # 512 pubkeys
    aggregate_pubkey: BLSPubkey  # Precomputed for efficiency

The aggregate_pubkey is a BLS aggregation of all 512 pubkeys. When all members participate, verification uses this single key instead of aggregating 512 keys on the fly.

SyncAggregate

What goes in each block:

class SyncAggregate(Container):
    sync_committee_bits: Bitvector[SYNC_COMMITTEE_SIZE]  # Who participated
    sync_committee_signature: BLSSignature  # Aggregated sig

The committee signs the previous slot's block root. This 1-slot delay gives time for block propagation before sync committee members need to sign.

🔧 Selection Algorithm

get_next_sync_committee_indices selects committee members using a similar pattern to proposer selection:

def get_next_sync_committee_indices(state: BeaconState) -> Sequence[ValidatorIndex]:
    epoch = Epoch(get_current_epoch(state) + 1)
    MAX_RANDOM_BYTE = 2**8 - 1
    active_validator_indices = get_active_validator_indices(state, epoch)
    active_validator_count = uint64(len(active_validator_indices))
    seed = get_seed(state, epoch, DOMAIN_SYNC_COMMITTEE)
    i = 0
    sync_committee_indices: List[ValidatorIndex] = []
    
    while len(sync_committee_indices) < SYNC_COMMITTEE_SIZE:
        shuffled_index = compute_shuffled_index(
            uint64(i % active_validator_count), active_validator_count, seed
        )
        candidate_index = active_validator_indices[shuffled_index]
        random_byte = hash(seed + uint_to_bytes(uint64(i // 32)))[i % 32]
        effective_balance = state.validators[candidate_index].effective_balance
        
        # Proportional selection: higher stake = more likely
        if effective_balance * MAX_RANDOM_BYTE >= MAX_EFFECTIVE_BALANCE * random_byte:
            sync_committee_indices.append(candidate_index)
        i += 1
    
    return sync_committee_indices
Critical detail: Unlike beacon committees, sync committee members can appear multiple times. With 500k validators competing for 512 spots over 27 hours, duplicates are rare but allowed. A validator appearing twice signs twice and gets double rewards.

Why balance weighting? Same rationale as proposer selection—stake-weighted selection maintains economic alignment. A 64 ETH validator should have twice the sync committee presence of a 32 ETH validator.

⚙️ Processing: process_sync_aggregate

The signature verification has a clever optimization:

if sum(committee_bits) == SYNC_COMMITTEE_SIZE:
    # All 512 participated: use precomputed aggregate key
    participant_pubkeys = [state.current_sync_committee.aggregate_pubkey]
    
elif sum(committee_bits) > SYNC_COMMITTEE_SIZE // 2:
    # >50% participated: subtract non-participants
    non_participant_aggregate = eth_aggregate_pubkeys(non_participant_pubkeys)
    participant_pubkey = bls.add(
        bls.bytes48_to_G1(aggregate_pubkey),
        bls.neg(bls.bytes48_to_G1(non_participant_aggregate))
    )
    participant_pubkeys = [BLSPubkey(bls.G1_to_bytes48(participant_pubkey))]
    
else:
    # <50%: aggregate participant keys directly
    participant_pubkeys = [pubkey for pubkey, bit in zip(...) if bit]

When >50% participate, it's cheaper to subtract non-participants from the precomputed aggregate than to aggregate participants from scratch. This is BLS math: agg(P1...Pn) - agg(Pm...Pk) = agg(remaining).

💰 Rewards & Penalties

Sync committee rewards use a dedicated weight:

SYNC_REWARD_WEIGHT = 2  (out of WEIGHT_DENOMINATOR = 64)

That's 2/64 = 3.125% of total validator rewards allocated to sync committee duties.

max_participant_rewards = Gwei(
    total_base_rewards * SYNC_REWARD_WEIGHT // WEIGHT_DENOMINATOR // SLOTS_PER_EPOCH
)
participant_reward = Gwei(max_participant_rewards // SYNC_COMMITTEE_SIZE)

The reward is divided by SLOTS_PER_EPOCH because sync committee members sign every slot for 256 epochs. Over the full period, this adds up.

Proposer Bonus

proposer_reward = Gwei(
    participant_reward * PROPOSER_WEIGHT // (WEIGHT_DENOMINATOR - PROPOSER_WEIGHT)
)

Penalties

Missing sync committee duties costs you the same as the reward. If you don't sign, you're penalized participant_reward per slot. Unlike attestation penalties, this is symmetric—miss a slot, lose exactly what you'd have earned.

🔄 Period Transitions

Committees rotate at epoch boundaries where epoch % 256 == 0:

def process_sync_committee_updates(state: BeaconState) -> None:
    next_epoch = get_current_epoch(state) + Epoch(1)
    if next_epoch % EPOCHS_PER_SYNC_COMMITTEE_PERIOD == 0:
        state.current_sync_committee = state.next_sync_committee
        state.next_sync_committee = get_next_sync_committee(state)

Two Committees in State

  • current_sync_committee: Active for this period
  • next_sync_committee: Precomputed for next period

Why precompute? Light clients need the next committee before the transition. Without lookahead, there'd be a gap where light clients can't verify blocks at period boundaries.

The next committee is computed ~27 hours in advance—enough time for light clients to learn it via sync protocol updates.

❓ Design Rationale

Why 512 validators?

Security vs efficiency tradeoff. 512 provides:

  • ~341 honest validators needed for 2/3 supermajority
  • Low probability of adversary controlling majority with <1/3 total stake
  • Small enough for light clients to track (512 × 48 bytes = 24 KB per committee)

Why 256 epochs (~27 hours)?

  • Long enough that validators can prepare (sync to correct head)
  • Short enough for stake fluidity (exited validators don't stay forever)
  • Light clients update ~once per day

Why separate from attesters?

Attesters rotate every epoch and focus on fork choice. Sync committees:

  • Have a dedicated purpose: enabling light clients
  • Can optimize for this single task
  • Don't add load to attestation subnets

  • Light Client Sync Protocol (altair/light-client/sync-protocol.md): The consumer of sync committees. Uses LightClientUpdate containing SyncAggregate to prove finalized headers.
  • Proposer Selection: Same balance-weighted random sampling pattern
  • Committee Index Aggregation (EIP-7549): Different problem—that's about attestation efficiency, not light clients

⚠️ Gotchas

  1. Duplicate membership: The same validator can appear multiple times in a sync committee. Check for this when analyzing participation rates.
  2. Signs previous slot: sync_aggregate in block N contains signatures over block N-1's root. This 1-slot delay is intentional.
  3. Symmetric penalties: Unlike attestation penalties (which only apply to source/target misses, not head), sync committee penalties are symmetric—miss = lose what you'd earn.
  4. Period boundary stakes: Light clients need next_sync_committee BEFORE the transition. The proof comes from the attested_header.state_root in LightClientUpdate.
