📚 Lesson 22: Execution Layer Requests (Electra)

From gossip to smart contracts — the unification of validator operations.

🔍 Spec Location

electra/beacon-chain.md → ExecutionRequests, process_operations

💡 Core Concept

Electra introduces a paradigm shift: validator operations (deposits, withdrawals, consolidations) now flow from the execution layer via smart contracts, rather than through the gossip network.

This is enabled by three EIPs working together:

  • EIP-6110 On-chain deposit supply (deposits via EL)
  • EIP-7002 Execution-layer triggerable exits/withdrawals
  • EIP-7251 MaxEB enables consolidations

The unifying structure is ExecutionRequests:

class ExecutionRequests(Container):
    deposits: List[DepositRequest, MAX_DEPOSIT_REQUESTS_PER_PAYLOAD]        # 8,192 max
    withdrawals: List[WithdrawalRequest, MAX_WITHDRAWAL_REQUESTS_PER_PAYLOAD]  # 16 max
    consolidations: List[ConsolidationRequest, MAX_CONSOLIDATION_REQUESTS_PER_PAYLOAD]  # 2 max

🛠 The Three Request Types

1. DepositRequest (EIP-6110)

class DepositRequest(Container):
    pubkey: BLSPubkey
    withdrawal_credentials: Bytes32
    amount: Gwei
    signature: BLSSignature
    index: uint64  # Sequential index from deposit contract

Why it matters: Eliminates the Eth1 follow distance delay (~13 hours). Deposits appear in the same block where the EL transaction was included.

2. WithdrawalRequest (EIP-7002)

class WithdrawalRequest(Container):
    source_address: ExecutionAddress  # Must match withdrawal credentials[12:]
    validator_pubkey: BLSPubkey
    amount: Gwei  # 0 = full exit, >0 = partial withdrawal

The magic constant: FULL_EXIT_REQUEST_AMOUNT = 0 signals a full exit.

3. ConsolidationRequest (EIP-7251)

class ConsolidationRequest(Container):
    source_address: ExecutionAddress
    source_pubkey: BLSPubkey
    target_pubkey: BLSPubkey

When source_pubkey == target_pubkey, it's a "switch to compounding" request (0x01 → 0x02 credentials).

🔗 The EL↔CL Flow

EL Smart Contract → ExecutionPayload → Engine API → CL process_operations
  1. User calls system contract on EL (deposit/withdrawal/consolidation)
  2. EL includes requests in block via EIP-7685 encoding
  3. Engine API's NewPayloadRequest includes execution_requests
  4. CL validates block hash includes requests list
  5. CL processes each request type in process_operations

The Eth1 Bridge Sunset

def process_operations(state, body):
    # Disable former deposit mechanism once all prior deposits processed
    eth1_deposit_index_limit = min(
        state.eth1_data.deposit_count,
        state.deposit_requests_start_index
    )
    if state.eth1_deposit_index < eth1_deposit_index_limit:
        assert len(body.deposits) == min(
            MAX_DEPOSITS, eth1_deposit_index_limit - state.eth1_deposit_index
        )
    else:
        assert len(body.deposits) == 0  # Old deposits disabled!

Once deposit_requests_start_index is set (on first EIP-6110 deposit), the old deposits field gradually drains to zero.

🛠 Processing Logic Deep Dive

process_deposit_request

def process_deposit_request(state, deposit_request):
    # Set start index on first deposit request ever
    if state.deposit_requests_start_index == UNSET_DEPOSIT_REQUESTS_START_INDEX:
        state.deposit_requests_start_index = deposit_request.index
    
    # Queue it for epoch processing
    state.pending_deposits.append(
        PendingDeposit(
            pubkey=deposit_request.pubkey,
            withdrawal_credentials=deposit_request.withdrawal_credentials,
            amount=deposit_request.amount,
            signature=deposit_request.signature,
            slot=state.slot,  # Non-GENESIS_SLOT marks it as EL request
        )
    )

No signature verification here! That happens in process_pending_deposits at epoch boundary.

process_withdrawal_request

def process_withdrawal_request(state, withdrawal_request):
    amount = withdrawal_request.amount
    is_full_exit = amount == FULL_EXIT_REQUEST_AMOUNT  # amount == 0
    
    # Verify: pubkey exists, credential match, active, not exiting
    # ...
    
    if is_full_exit:
        if pending_balance_to_withdraw == 0:
            initiate_validator_exit(state, index)
        return
    
    # Partial withdrawal: requires compounding credentials (0x02)
    if has_compounding_withdrawal_credential(validator):
        to_withdraw = min(
            state.balances[index] - MIN_ACTIVATION_BALANCE - pending_balance,
            amount
        )
        exit_queue_epoch = compute_exit_epoch_and_update_churn(state, to_withdraw)
        state.pending_partial_withdrawals.append(
            PendingPartialWithdrawal(
                validator_index=index,
                amount=to_withdraw,
                withdrawable_epoch=exit_queue_epoch + MIN_VALIDATOR_WITHDRAWABILITY_DELAY,
            )
        )

Key insight: Partial withdrawals consume churn just like exits. Why? Because withdrawing 100 ETH from a 2048 ETH validator affects the stake distribution.

process_consolidation_request — Two Modes

def process_consolidation_request(state, consolidation_request):
    # Mode 1: Switch to compounding (source == target)
    if is_valid_switch_to_compounding_request(state, consolidation_request):
        switch_to_compounding_validator(state, source_index)
        return
    
    # Mode 2: Actual consolidation (source != target)
    # - Target must have compounding credentials (0x02)
    # - Source exits and pending consolidation queued
    source_validator.exit_epoch = compute_consolidation_epoch_and_update_churn(
        state, source_validator.effective_balance
    )
    state.pending_consolidations.append(
        PendingConsolidation(source_index, target_index)
    )

⚠️ Gotchas

  1. Authorization model: For withdrawals/consolidations, source_address must match withdrawal_credentials[12:]. The EL contract verifies msg.sender.
  2. Pending balance check: Can't exit if get_pending_balance_to_withdraw(state, index) > 0. Must wait for pending partials to clear.
  3. Churn competition: Exits, partial withdrawals, and consolidations all compete for churn capacity. Consolidations have their own limit: get_consolidation_churn_limit() = get_balance_churn_limit() - get_activation_exit_churn_limit().
  4. Queue overflow: If pending_partial_withdrawals is full (2^27 entries), partial withdrawal requests are silently ignored. Only full exits proceed.

❓ Why This Way

The old model required:

  • Gossip network for voluntary exits (unreliable)
  • ~13 hour eth1 follow distance for deposits (slow)
  • No way to trigger exits from smart contracts (inflexible)

The new model enables:

  • Staking pools with contract-controlled exits
  • Instant deposit visibility
  • Unified security model via EL transaction authorization
  • Future composability (restaking protocols can trigger exits)
