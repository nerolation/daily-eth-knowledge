💡 Core Concept

AttestationData is the atomic unit of consensus participation. Every validator's duty is to produce this 5-field structure once per epoch. It encodes two distinct votes:

  1. LMD-GHOST vote (head vote) → beacon_block_root
  2. FFG vote (finality vote) → source + target

The brilliance: both votes share the same BLS signature, so you get fork choice AND finality participation in one 112-byte commitment.

class AttestationData(Container):
    slot: Slot
    index: CommitteeIndex
    beacon_block_root: Root  # LMD-GHOST vote
    source: Checkpoint       # FFG source
    target: Checkpoint       # FFG target

Each Checkpoint is just:

class Checkpoint(Container):
    epoch: Epoch
    root: Root

🛠 Field-by-Field Analysis

1. slot: Slot

When the attestation was created. NOT when it's included on-chain.

  • Used for: committee assignment, inclusion delay calculation, timing validation
  • Constraint: data.slot + 1 <= state.slot <= data.slot + 32

2. index: CommitteeIndex

Which of the slot's committees produced this attestation.

  • Valid range: 0 <= index < get_committee_count_per_slot(state, epoch)
  • With ~1M validators: typically 64 committees per slot (MAX_COMMITTEES_PER_SLOT)
  • With fewer validators: scales down but each committee stays ≥128 (TARGET_COMMITTEE_SIZE)

3. beacon_block_root: Root

The LMD-GHOST vote. Points to the block the attester considers the head.

  • Fork choice uses this via update_latest_messages():
store.latest_messages[i] = LatestMessage(
    epoch=target.epoch, 
    root=beacon_block_root  # ← this is what counts
)
  • Must point to a known block at or before slot

4. source: Checkpoint

The FFG source—the checkpoint the attester is voting FROM.

  • Must match state.current_justified_checkpoint or state.previous_justified_checkpoint
if data.target.epoch == get_current_epoch(state):
    assert data.source == state.current_justified_checkpoint
else:
    assert data.source == state.previous_justified_checkpoint

5. target: Checkpoint

The FFG target—the checkpoint the attester is voting TO justify.

  • target.epoch must equal compute_epoch_at_slot(data.slot) — you vote for YOUR epoch
  • target.root is the block root at the first slot of the target epoch

❓ Why This Structure?

The split between LMD-GHOST and FFG is fundamental to Ethereum's consensus:

Component Purpose What it affects
beacon_block_root "Which chain do I think is canonical?" Fork choice (get_head)
source → target "I want to finalize this checkpoint" Justification/finalization

Critical insight: beacon_block_root and target.root are NOT necessarily the same block.

  • beacon_block_root: any block at or before slot
  • target.root: specifically the block at epoch boundary

Fork choice validation ensures consistency:

# LMD vote must be consistent with FFG vote target
assert target.root == get_checkpoint_block(
    store, attestation.data.beacon_block_root, target.epoch
)

Translation: "Your head vote must be a descendant of your target checkpoint."

🔗 From AttestationData to On-Chain

AttestationData gets wrapped in multiple containers depending on context:

AttestationData (just the vote)
     ↓
Attestation (+ aggregation_bits + signature)
     ↓
IndexedAttestation (+ explicit validator indices, sorted)

The aggregation_bits bitlist maps to committee members:

def get_attesting_indices(state, attestation):
    committee = get_beacon_committee(state, data.slot, data.index)
    return set(
        index for i, index in enumerate(committee) 
        if attestation.aggregation_bits[i]
    )

⚠️ Gotchas

1. Signature domain uses TARGET epoch 
domain = get_domain(state, DOMAIN_BEACON_ATTESTER, 
                    indexed_attestation.data.target.epoch)

Not slot. This prevents cross-epoch signature replay.

2. Epoch boundary attestations

If you attest in slot 32 (first slot of epoch 1), your target is epoch 1, but your source might be the epoch 0 justified checkpoint. The "previous justified" rule exists because justification lags.

3. Empty slot edge case

If slot N had no block, an attestation at slot N still has a valid beacon_block_root—it just points to the most recent block (from slot N-1 or earlier).

4. Committee index change (EIP-7549, Electra)

In Electra, the index field remains in AttestationData but is constrained to 0 (assert data.index == 0). Committee membership moves to a new committee_bits bitvector in the Attestation container. This enables cross-committee aggregation—attestations with identical slot, beacon_block_root, source, target can now aggregate regardless of committee.

📖 Summary

AttestationData = 2 votes, 1 signature:

  • Head vote: "I think block X is canonical"
  • FFG vote: "I support finalizing checkpoint C"
