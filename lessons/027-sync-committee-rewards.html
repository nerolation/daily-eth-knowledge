📚 Lesson 27: Sync Committee Rewards

How validators get paid for enabling light clients

🔍 Spec Location

altair/beacon-chain.md → process_sync_aggregate, incentivization weights

💡 Core Concept

Sync committee members sign the previous slot's block root every slot. For this service, they receive SYNC_REWARD_WEIGHT = 2 out of 64 total weight—about 3.1% of total issuance. It's a small slice, but sync committee duty is also light: sign one thing per slot for ~27 hours.

The design is deliberately symmetric: if you participate, you earn the reward. If you miss, you lose the same amount. This makes the expected value exactly zero for a validator who randomly participates 50% of the time—participation only pays if you're actually reliable.

🛠 The Reward Math

From process_sync_aggregate:

# Compute participant and proposer rewards
total_active_increments = get_total_active_balance(state) // EFFECTIVE_BALANCE_INCREMENT
total_base_rewards = Gwei(get_base_reward_per_increment(state) * total_active_increments)
max_participant_rewards = Gwei(
    total_base_rewards * SYNC_REWARD_WEIGHT // WEIGHT_DENOMINATOR // SLOTS_PER_EPOCH
)
participant_reward = Gwei(max_participant_rewards // SYNC_COMMITTEE_SIZE)
proposer_reward = Gwei(
    participant_reward * PROPOSER_WEIGHT // (WEIGHT_DENOMINATOR - PROPOSER_WEIGHT)
)

Breaking this down:

  1. total_base_rewards = what all validators would earn in total per epoch
  2. max_participant_rewards = sync portion (2/64) divided by 32 slots
  3. participant_reward = that, split among 512 members
  4. proposer_reward = proposer gets their cut (8/56 of participant_reward)

The proposer is incentivized to include sync aggregates because they earn ~14% extra for each participating member.

🛠 Symmetric Application

# Apply participant and proposer rewards
for participant_index, participation_bit in zip(
    committee_indices, sync_aggregate.sync_committee_bits
):
    if participation_bit:
        increase_balance(state, participant_index, participant_reward)
        increase_balance(state, get_beacon_proposer_index(state), proposer_reward)
    else:
        decrease_balance(state, participant_index, participant_reward)

Key insight: Non-participants lose participant_reward—the same amount participants gain. No proposer reward flows when someone misses.

Why symmetric?

  • Makes sync duty "opt-in profitable": you only benefit if you reliably participate
  • Discourages validators from ignoring sync duty during committee membership
  • Simple mental model: +R for showing up, -R for not

This differs from attestation rewards where penalties are proportional to weight but not exactly symmetric.

❓ Why This Way

Why only 2/64 weight?

Sync committees serve light clients, not consensus safety. Attestations (source/target/head) secure finality; sync signatures just make it cheaply verifiable. Lower reward = appropriate for lower-stakes duty.

Why per-slot rewards instead of epoch?

Sync signatures are processed every block, not at epoch boundaries. Immediate feedback loop: miss a slot, lose that slot's reward. No batching or averaging.

Why proposer cut?

Proposers must include SyncAggregate in blocks. Without a reward, they might skip it (saves bytes, computation). The ~14% cut ensures proposers want sync aggregates included.

⚠️ Gotchas

1. Duplicates in committee
A validator can appear multiple times in the 512-member committee (weighted selection allows this). They earn/lose per occurrence.
2. No inactivity leak for sync
Missing sync duty triggers symmetric penalty, but not the inactivity leak. Leak only applies to attestation target misses.
3. Precomputed aggregate key optimization
If all 512 participate, the spec uses the precomputed aggregate_pubkey instead of aggregating 512 keys—significant gas savings in the common case.
