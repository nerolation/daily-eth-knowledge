Date: February 23, 2026

💡 Why Consolidations Exist

Consolidations are the core mechanism enabling EIP-7251 (MaxEB). Without them, operators running multiple 32 ETH validators would have no way to merge them into larger validators that can accumulate up to 2048 ETH effective balance.

The consolidation mechanism serves two purposes:

  1. Validator merging: Move stake from a source validator to a target validator
  2. Credential upgrade: Switch from 0x01 (ETH1) to 0x02 (compounding) withdrawal credentials
Key insight: Only validators with 0x02 (compounding) credentials can have effective balance above 32 ETH. Consolidation is how you get there.

🔍 The Request Flow

Consolidations are execution-layer triggered (EIP-7251). The flow:

EL Contract → ExecutionPayload.execution_requests.consolidations
    → CL process_consolidation_request()
    → state.pending_consolidations queue
    → process_pending_consolidations() at epoch boundary
    → Balance transfer complete

ConsolidationRequest Container

class ConsolidationRequest(Container):
    source_address: ExecutionAddress  # Must match source's withdrawal creds
    source_pubkey: BLSPubkey         # Validator to exit
    target_pubkey: BLSPubkey         # Validator receiving the balance

The source_address must match the withdrawal credentials of the source validator—this proves authorization. Only the withdrawal address owner can consolidate their validator.

PendingConsolidation Container

class PendingConsolidation(Container):
    source_index: ValidatorIndex
    target_index: ValidatorIndex

Simple pair of indices. The balance amount isn't stored—it's computed at processing time from the source's current state.

🛠 Consolidation Churn

Consolidations have their own separate churn limit, distinct from activation/exit churn:

def get_consolidation_churn_limit(state: BeaconState) -> Gwei:
    return get_balance_churn_limit(state) - get_activation_exit_churn_limit(state)

This means:

  • Total balance churn = max(128 ETH, total_active_balance / 65536)
  • Activation/Exit churn = min(256 ETH, total_balance_churn)
  • Consolidation churn = the remainder
Total Active Balance Balance Churn Exit Churn Consolidation Churn
<8.4M ETH 128 ETH 128 ETH 0 ETH (blocked!)
16.7M ETH 256 ETH 256 ETH 0 ETH (blocked!)
33.5M ETH (current) 512 ETH 256 ETH 256 ETH
⚠️ Critical edge case: If get_consolidation_churn_limit(state) <= MIN_ACTIVATION_BALANCE, consolidation requests are silently ignored! This protects against consolidations when the network is too small.

🔧 process_consolidation_request

The core processing function handles two cases:

Case 1: Switch to Compounding (source == target)

if is_valid_switch_to_compounding_request(state, consolidation_request):
    # Just upgrade credentials, no balance movement
    switch_to_compounding_validator(state, source_index)
    return

What switch_to_compounding_validator does:

def switch_to_compounding_validator(state: BeaconState, index: ValidatorIndex) -> None:
    validator = state.validators[index]
    # Change 0x01... to 0x02...
    validator.withdrawal_credentials = (
        COMPOUNDING_WITHDRAWAL_PREFIX + validator.withdrawal_credentials[1:]
    )
    queue_excess_active_balance(state, index)

The queue_excess_active_balance is subtle: it takes any balance above 32 ETH and queues it as a pending deposit back to the same validator. This handles the case where you had 33 ETH before switching—that extra 1 ETH needs to be "re-deposited" to count toward your new higher effective balance.

Case 2: Full Consolidation (source → target)

Validation checks for full consolidation:

# Source must have execution credentials (0x01 or 0x02)
has_correct_credential = has_execution_withdrawal_credential(source_validator)
# Authorization: source address must match
is_correct_source_address = (
    source_validator.withdrawal_credentials[12:] == consolidation_request.source_address
)

# Target MUST have compounding credentials (0x02)
if not has_compounding_withdrawal_credential(target_validator):
    return

# Both must be active, neither can be exiting
if source_validator.exit_epoch != FAR_FUTURE_EPOCH:
    return
if target_validator.exit_epoch != FAR_FUTURE_EPOCH:
    return

# Source must have been active for SHARD_COMMITTEE_PERIOD (~27 hours)
if current_epoch < source_validator.activation_epoch + SHARD_COMMITTEE_PERIOD:
    return

# No pending withdrawals allowed on source
if get_pending_balance_to_withdraw(state, source_index) > 0:
    return

If all checks pass:

# Initiate exit for source (uses consolidation churn, not exit churn!)
source_validator.exit_epoch = compute_consolidation_epoch_and_update_churn(
    state, source_validator.effective_balance
)
source_validator.withdrawable_epoch = Epoch(
    source_validator.exit_epoch + MIN_VALIDATOR_WITHDRAWABILITY_DELAY
)

# Queue the consolidation for epoch processing
state.pending_consolidations.append(
    PendingConsolidation(source_index=source_index, target_index=target_index)
)

⏰ Epoch Processing

At each epoch boundary, process_pending_consolidations handles ready consolidations:

def process_pending_consolidations(state: BeaconState) -> None:
    next_epoch = Epoch(get_current_epoch(state) + 1)
    next_pending_consolidation = 0
    
    for pending_consolidation in state.pending_consolidations:
        source_validator = state.validators[pending_consolidation.source_index]
        
        # Skip slashed sources (they can't consolidate)
        if source_validator.slashed:
            next_pending_consolidation += 1
            continue
        
        # Wait until source is withdrawable
        if source_validator.withdrawable_epoch > next_epoch:
            break
        
        # Calculate what to move (min of balance and effective_balance)
        source_effective_balance = min(
            state.balances[pending_consolidation.source_index],
            source_validator.effective_balance
        )
        
        # Do the transfer
        decrease_balance(state, pending_consolidation.source_index, source_effective_balance)
        increase_balance(state, pending_consolidation.target_index, source_effective_balance)
        
        next_pending_consolidation += 1
    
    # Remove processed consolidations
    state.pending_consolidations = state.pending_consolidations[next_pending_consolidation:]

❓ Why This Design?

Why separate consolidation churn?

If consolidations shared the exit queue, a coordinated consolidation event could delay legitimate exits. Separate churn ensures consolidations (an optimization) can't interfere with exits (a security mechanism).

Why target must be 0x02?

If target had 0x01 credentials, the received balance above 32 ETH would be immediately withdrawable but not stakeable. The consolidation would effectively become a partial exit. Requiring 0x02 ensures the balance actually compounds.

Why wait for withdrawable_epoch?

The source validator is exiting. If we moved funds immediately, the source could be slashed between consolidation request and actual exit, creating accounting issues. Waiting until withdrawable_epoch ensures all slashing windows have passed.

Why no pending withdrawals on source?

Pending partial withdrawals create a commitment about future balance. Allowing consolidation with pending withdrawals would create conflicting claims on the same funds.

⚠️ Gotchas

  • Consolidation churn can be zero if total stake is low—requests silently fail
  • Slashed sources have their consolidation skipped but still removed from queue
  • The balance transferred is min(balance, effective_balance), not just balance
  • 0x00 (BLS) credentials cannot consolidate—must switch to 0x01 first via BLSToExecutionChange
  • Queue limit: PENDING_CONSOLIDATIONS_LIMIT = 262,144—requests fail when full

📖 Summary

Consolidations enable the transition to MaxEB by providing two operations:

  1. Credential switch: 0x01 → 0x02 (same validator, enables compounding)
  2. Validator merge: Exit source, transfer balance to compounding target

The mechanism uses separate churn to avoid impacting exits, requires target to be 0x02 to ensure funds compound, and waits for the source's withdrawable epoch before transferring to handle slashing edge cases correctly.
