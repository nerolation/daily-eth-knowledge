Lesson 21 February 24, 2026 Module 2: Validator Lifecycle

MaxEB - Variable Effective Balance

EIP-7251 | Electra Fork

📚 Overview

Topic: Variable effective balance through compounding credentials

Spec Location: electra/beacon-chain.md → Gwei values, get_max_effective_balance, switch_to_compounding_validator

💡 Core Concept

Before Electra, every validator was capped at 32 ETH effective balance. Period. Whether you deposited 32 ETH or 2000 ETH, your voting power was the same.

This was... wasteful.

EIP-7251 introduces variable effective balance through a new withdrawal credential type:

Credential Prefix Name Max Effective Balance
0x00 BLS withdrawal 32 ETH
0x01 Execution address 32 ETH
0x02 Compounding 2048 ETH

The key insight: validators with 0x02 credentials can compound their rewards rather than auto-withdrawing excess, and can scale up to 2048 ETH.

🛠 The Core Constants

# Electra preset - Gwei values
MIN_ACTIVATION_BALANCE = Gwei(2**5 * 10**9)   # 32 ETH
MAX_EFFECTIVE_BALANCE_ELECTRA = Gwei(2**11 * 10**9)  # 2048 ETH

# New withdrawal prefix
COMPOUNDING_WITHDRAWAL_PREFIX = Bytes1('0x02')

Notice the split: 32 ETH to activate, but up to 2048 ETH effective balance for compounding validators. That's 64x the previous cap.

🔧 The Key Helper: get_max_effective_balance

This function is the heart of MaxEB - it determines each validator's cap:

def get_max_effective_balance(validator: Validator) -> Gwei:
    """
    Get max effective balance for ``validator``.
    """
    if has_compounding_withdrawal_credential(validator):
        return MAX_EFFECTIVE_BALANCE_ELECTRA  # 2048 ETH
    else:
        return MIN_ACTIVATION_BALANCE  # 32 ETH

The credential check:

def has_compounding_withdrawal_credential(validator: Validator) -> bool:
    return is_compounding_withdrawal_credential(
        validator.withdrawal_credentials
    )

def is_compounding_withdrawal_credential(
    withdrawal_credentials: Bytes32
) -> bool:
    return withdrawal_credentials[:1] == COMPOUNDING_WITHDRAWAL_PREFIX

One byte prefix determines your max stake.

🔄 Switching to Compounding

Validators can upgrade their credentials from 0x01 to 0x02:

def switch_to_compounding_validator(
    state: BeaconState, index: ValidatorIndex
) -> None:
    validator = state.validators[index]
    validator.withdrawal_credentials = (
        COMPOUNDING_WITHDRAWAL_PREFIX  # 0x02
        + validator.withdrawal_credentials[1:]  # Keep the address
    )
    queue_excess_active_balance(state, index)

The tricky part: when switching, any balance above 32 ETH gets queued as a pending deposit:

def queue_excess_active_balance(
    state: BeaconState, index: ValidatorIndex
) -> None:
    balance = state.balances[index]
    if balance > MIN_ACTIVATION_BALANCE:  # > 32 ETH
        excess_balance = balance - MIN_ACTIVATION_BALANCE
        state.balances[index] = MIN_ACTIVATION_BALANCE
        # Queue excess as pending deposit
        state.pending_deposits.append(
            PendingDeposit(
                pubkey=validator.pubkey,
                withdrawal_credentials=validator.withdrawal_credentials,
                amount=excess_balance,
                signature=bls.G2_POINT_AT_INFINITY,
                slot=GENESIS_SLOT,  # Marker for "not a deposit request"
            )
        )

Why? The excess needs to go through the deposit churn limit before becoming effective. Can't just instantly 64x your voting power.

⚖️ Proportional Weight in Selection

MaxEB validators get proportionally more chances to propose and serve on sync committees. The selection algorithms were updated:

Proposer Selection (Modified)

def compute_proposer_index(state, indices, seed) -> ValidatorIndex:
    MAX_RANDOM_VALUE = 2**16 - 1  # Was 2**8-1 (8-bit)
    i = uint64(0)
    while True:
        candidate_index = indices[compute_shuffled_index(i % total, total, seed)]
        # Now uses 16-bit random instead of 8-bit
        random_bytes = hash(seed + uint_to_bytes(i // 16))
        offset = i % 16 * 2
        random_value = bytes_to_uint64(random_bytes[offset : offset + 2])
        effective_balance = state.validators[candidate_index].effective_balance
        
        # Key change: compare against MAX_EFFECTIVE_BALANCE_ELECTRA
        if effective_balance * MAX_RANDOM_VALUE >= \
           MAX_EFFECTIVE_BALANCE_ELECTRA * random_value:
            return candidate_index
        i += 1

Why 16-bit instead of 8-bit? With 2048 ETH max balance (64x larger range), 8 bits of randomness isn't granular enough. 16 bits provides sufficient precision.

A 2048 ETH validator will be selected as proposer ~64x more often than a 32 ETH validator. Proportional rewards for proportional stake.

📊 Effective Balance Updates

The hysteresis logic now respects variable caps:

def process_effective_balance_updates(state: BeaconState) -> None:
    for index, validator in enumerate(state.validators):
        balance = state.balances[index]
        # Get the cap for THIS validator
        max_effective_balance = get_max_effective_balance(validator)
        
        # Same hysteresis logic, but with per-validator cap
        if (balance + DOWNWARD_THRESHOLD < validator.effective_balance
            or validator.effective_balance + UPWARD_THRESHOLD < balance):
            validator.effective_balance = min(
                balance - balance % EFFECTIVE_BALANCE_INCREMENT,
                max_effective_balance  # 32 or 2048 ETH
            )

1 ETH granularity still applies. A 2050 ETH balance → 2048 ETH effective balance.

❓ Why This Way?

The state bloat problem:

  • Pre-Electra: ~1M validators, each with a Validator record
  • Large stakers (exchanges, institutions) run hundreds of validators
  • Each validator = ~121 bytes of state + index overhead + participation tracking

The solution:

  • Allow consolidation: 64 × 32 ETH validators → 1 × 2048 ETH validator
  • Same economic security, 64x less state
  • Reduces validator set size without reducing stake

Why 2048 ETH max?

  • 64x factor = clean power of 2
  • Balances protocol complexity vs. state reduction benefits
  • Higher would concentrate proposer selection too much
  • Lower wouldn't solve the problem enough

Why require 0x02 opt-in?

  • Not everyone wants rewards to compound
  • Some want regular ETH income (staking-as-a-service)
  • Clean separation: 0x01 = auto-withdraw excess, 0x02 = compound

⚠️ Gotchas

  1. MIN_ACTIVATION_BALANCE ≠ MAX_EFFECTIVE_BALANCE anymore
    • Phase0: both were 32 ETH
    • Electra: 32 ETH to activate, up to 2048 ETH effective
  2. Switching is one-way: 0x01 → 0x02 only, no going back
  3. Excess gets queued: When switching, your balance above 32 ETH doesn't instantly count. Goes through pending deposits.
  4. Slashing scales with balance: A 2048 ETH validator can lose up to 2048/4096 = 0.5 ETH initial penalty (vs 32/4096 for a 32 ETH validator)
  5. Inactivity leak scales too: Offline with 2048 ETH hurts 64x more in absolute terms
