Lesson 24 Module 3: Committees and Shuffling

Beacon Committees

How validators are assigned to attest each slot

📚 Overview

In the previous lesson, we learned how compute_shuffled_index creates an unpredictable permutation of validators. Today we see how that shuffle becomes committee assignments.

Spec Reference: phase0/beacon-chain.md → get_beacon_committee, get_committee_count_per_slot

🎯 Core Problem

Every slot, validators need to attest to the head of the chain. But with ~1M validators, you can't have everyone attest to everything. The solution: divide validators into committees—small groups that attest together.

The committee system must satisfy:

  1. Coverage: Every active validator attests exactly once per epoch
  2. Security: Each committee has enough validators that controlling it is infeasible
  3. Efficiency: Committee sizes are bounded so aggregation is practical
  4. Unpredictability: Assignments can't be known far in advance

🔢 The Constants

TARGET_COMMITTEE_SIZE    = 128    # Minimum safe committee size
MAX_COMMITTEES_PER_SLOT  = 64     # Upper bound on parallelism
SLOTS_PER_EPOCH          = 32     # Committees rotate each epoch

Why 128? From the spec:

For the safety of committees, TARGET_COMMITTEE_SIZE exceeds the recommended minimum committee size of 111

The 111 comes from birthday-bound analysis—below this threshold, an attacker with <1/3 stake has non-negligible probability of controlling a committee by chance.

📐 How Many Committees Per Slot?

def get_committee_count_per_slot(state: BeaconState, epoch: Epoch) -> uint64:
    return max(
        uint64(1),
        min(
            MAX_COMMITTEES_PER_SLOT,  # 64
            uint64(len(get_active_validator_indices(state, epoch)))
            // SLOTS_PER_EPOCH        # 32
            // TARGET_COMMITTEE_SIZE, # 128
        ),
    )

The Math

The formula is: active_validators / 32 / 128 = active_validators / 4096

Active Validators Calculation Result
1,000,000 1,000,000 / 4096 ≈ 244 64 (capped)
400,000 400,000 / 4096 ≈ 97 64 (capped)
100,000 100,000 / 4096 ≈ 24 24
4,000 4,000 / 4096 < 1 1 (minimum)
Key Insight: The formula ensures committee sizes stay ≥128 whenever possible. More validators = more parallel committees, NOT larger committees.

Total committees per epoch: committees_per_slot × 32

  • At 64 committees/slot: 2048 committees/epoch
  • Each validator appears in exactly one committee per epoch

🔧 Building the Committee

def get_beacon_committee(
    state: BeaconState, slot: Slot, index: CommitteeIndex
) -> Sequence[ValidatorIndex]:
    epoch = compute_epoch_at_slot(slot)
    committees_per_slot = get_committee_count_per_slot(state, epoch)
    return compute_committee(
        indices=get_active_validator_indices(state, epoch),
        seed=get_seed(state, epoch, DOMAIN_BEACON_ATTESTER),
        index=(slot % SLOTS_PER_EPOCH) * committees_per_slot + index,
        count=committees_per_slot * SLOTS_PER_EPOCH,
    )

The Index Calculation

The index calculation is crucial:

  • slot % SLOTS_PER_EPOCH → which slot within the epoch (0-31)
  • Multiply by committees_per_slot → offset for this slot
  • Add index → which committee within this slot

Total count = 64 × 32 = 2048 committees, and each validator lands in exactly one.

🎰 compute_committee: Where Shuffling Meets Slicing

def compute_committee(
    indices: Sequence[ValidatorIndex],
    seed: Bytes32,
    index: uint64,
    count: uint64,
) -> Sequence[ValidatorIndex]:
    start = (len(indices) * index) // count
    end = (len(indices) * (index + 1)) // count
    return [
        indices[compute_shuffled_index(uint64(i), uint64(len(indices)), seed)]
        for i in range(start, end)
    ]

The Algorithm

  1. Divide the validator set into count equal-ish slices
  2. For committee index, take validators from position start to end
  3. But don't take validators at those positions—take the shuffled positions
Why not shuffle first, then slice?

That would require shuffling the entire validator set (expensive). Instead: compute the shuffled index only for positions we need.

With 1M validators and 2048 committees: each committee ~488 validators. We only call compute_shuffled_index ~488 times, not 1M times.

The Seed

get_seed(state, epoch, DOMAIN_BEACON_ATTESTER)

  • Derived from RANDAO + epoch + domain
  • Same seed = same shuffle for entire epoch
  • Domain separation ensures attester shuffle ≠ proposer shuffle

⚔️ Security Analysis

Why TARGET_COMMITTEE_SIZE = 128?

With random sampling, probability that attacker with p stake controls ≥2/3 of a committee of size n:

P(attack) ≈ exp(-n × D(2/3 || p))

where D is KL divergence. For p=1/3 and n=128:

  • P(attack) ≈ 2-40 (one in a trillion)

For n=64: P(attack) ≈ 2-20 (plausible with many attempts)

The 111 threshold from Vitalik's analysis assumes slightly different parameters, but same principle: below ~100, random committee takeover becomes feasible.

What if not enough validators?

At genesis with only 16,384 validators:

  • committees_per_slot = 16384 / 4096 = 4
  • Committee size = 16384 / (4 × 32) = 128 ✓

The formula automatically reduces parallelism to maintain minimum committee size.

At the minimum (1 committee/slot)

  • 4096 validators or fewer
  • Each committee = all validators in that slot
  • Still rotates across the epoch

🔗 Committees → Attestations

When you attest, you include:

class AttestationData(Container):
    slot: Slot              # When you're attesting
    index: CommitteeIndex   # Which committee you're in
    beacon_block_root: Root # Your head vote
    source: Checkpoint      # FFG source
    target: Checkpoint      # FFG target

The index field identifies your committee. Your validator client:

  1. Calls get_beacon_committee(state, slot, index) to get committee members
  2. Finds your position in that list
  3. Sets the corresponding bit in aggregation_bits

Attestation aggregation happens within committees:

  • All committee members voting for the same head can be merged
  • One signature covers up to 2048 validators (MAX_VALIDATORS_PER_COMMITTEE)

⚠️ Gotchas

  1. Committees are epoch-stable: Same committee assignments for all 32 slots of an epoch (but each validator only attests once)
  2. Index bounds: data.index < get_committee_count_per_slot(state, epoch) is checked in attestation processing
  3. Empty slots: If a slot has no block, validators still attest. Committee assignments don't depend on block presence.
