üí° Core Concept

SSZ (Simple Serialize) is the canonical serialization format for all consensus layer data. Unlike RLP (used in the EL), SSZ was designed with three goals:

  1. Determinism ‚Äî identical objects always produce identical bytes
  2. Merkleization ‚Äî every object has a unique 32-byte hash_tree_root
  3. Proof-friendliness ‚Äî any field can be proven with O(log n) hashes
Key insight: SSZ doesn't just serialize data‚Äîit defines a canonical Merkle tree structure for every type. This is what enables light clients.

Containers are the workhorses. Every major consensus structure (BeaconState, BeaconBlock, Attestation) is a Container.

üõ† Serialization Rules

The serialization algorithm distinguishes fixed-size vs variable-size types.

  • Fixed-size: uints, booleans, Vectors of fixed types, Containers with only fixed fields
  • Variable-size: Lists, Bitlists, anything containing variable-size types

For fixed-size types, serialization is trivial:

# uint64 serialization
return value.to_bytes(8, "little")  # Always little-endian!

For variable-size containers, SSZ uses an offset table:

# Serializing Container with variable fields:
# 1. Fixed parts: either the value OR a 4-byte offset
# 2. Variable parts: actual variable-length data

fixed_parts = [serialize(x) if fixed else 4-byte-offset for x in fields]
variable_parts = [serialize(x) for x in variable_fields]
return concat(fixed_parts) + concat(variable_parts)

The offset tells you where in the byte stream the variable-length data starts. This enables random access to fixed fields without parsing variable fields.

Constants that matter:

BYTES_PER_CHUNK         = 32    # Merkle tree leaf size
BYTES_PER_LENGTH_OFFSET = 4     # Max object size: 2^32 bytes

The Rules (Simplified)

def hash_tree_root(value):
    if is_basic(value):
        return merkleize(pack(value))
    
    if is_vector_of_basic(value):
        return merkleize(pack(value))
    
    if is_list_of_basic(value):
        return mix_in_length(
            merkleize(pack(value), limit=chunk_count(type)),
            len(value)
        )
    
    if is_container(value):
        return merkleize([
            hash_tree_root(field) for field in value
        ])
    
    if is_list_of_composite(value):
        return mix_in_length(
            merkleize([hash_tree_root(x) for x in value], limit=N),
            len(value)
        )
Key insight: Lists mix in their length after merkleization. This means the tree structure is determined by the capacity (N), not the current length. Two lists with the same capacity but different lengths produce trees of the same depth‚Äîcritical for stable generalized indices.

üìç Generalized Indices

A generalized index is a way to address any node in an SSZ Merkle tree using a single integer.

Definition: generalized_index = 2^depth + index

       1           ‚Üê root (generalized index 1)
    2     3        ‚Üê depth 1
   4 5   6 7       ‚Üê depth 2
  ...

Properties

  • Root is always 1
  • Left child of node k is 2k
  • Right child of node k is 2k + 1
  • Parent of node k is k // 2
  • Sibling of node k is k ^ 1 (XOR with 1)

Computing the Generalized Index for a Path

def get_generalized_index(typ, *path):
    # path like [7, "foo", 3] for x[7].foo[3]
    root = 1
    for p in path:
        if p == "__len__":
            # Length is right child of list root
            root = root * 2 + 1
        else:
            pos = get_item_position(typ, p)
            base = 2 if is_list(typ) else 1  # Lists have data on left
            root = root * base * next_pow_of_two(chunk_count(typ)) + pos
            typ = get_elem_type(typ, p)
    return root

Why this matters: You can now refer to "slot 4 of the state's block_roots" as a single integer that's stable across all states of the same type.

üîê Merkle Proofs

From phase0/beacon-chain.md:

def is_valid_merkle_branch(
    leaf: Bytes32,
    branch: Sequence[Bytes32],
    depth: uint64,
    index: uint64,
    root: Root
) -> bool:
    value = leaf
    for i in range(depth):
        if index // (2**i) % 2:
            # We're on the right ‚Üí sibling on left
            value = hash(branch[i] + value)
        else:
            # We're on the left ‚Üí sibling on right
            value = hash(value + branch[i])
    return value == root

The index here uses the simpler "index at depth" convention (not generalized index), but the principle is the same: walk up the tree, hashing with siblings.

Multiproofs

More efficient when proving multiple leaves. From merkle-proofs.md:

def get_helper_indices(indices):
    """
    Get the minimal set of auxiliary nodes needed 
    to prove multiple generalized indices.
    """
    all_helper = set()
    all_path = set()
    for idx in indices:
        all_helper |= set(get_branch_indices(idx))
        all_path |= set(get_path_indices(idx))
    return sorted(all_helper - all_path, reverse=True)

A multiproof for k leaves at the same depth needs only ~k * log(n/k) auxiliary nodes, not k * log(n).

‚ùì Why SSZ Over RLP?

PropertyRLPSSZ
Merkle-nativeNoYes
Fixed offsetsNoYes (for fixed types)
Light client proofsExpensiveNative

RLP is fine for serialization, but it's opaque to Merkle operations. To prove a field in an RLP structure, you'd need custom logic per type. With SSZ, the proof format is universal.

‚ö†Ô∏è Gotchas

  1. Little-endian everywhere. All integers are serialized little-endian. If you're used to big-endian EL conventions, this will bite you.
  2. Capacity ‚â† length. A List[uint64, 1000] with 5 elements still has a Merkle tree sized for 1000 elements (padded with zeros). The actual length is mixed in separately.
  3. Container ordering matters. The Merkle tree structure depends on field order. Reordering fields changes the hash_tree_root.
  4. Bitvector vs List[boolean]. Both are valid but serialize differently. Bitvector is packed (8 bits per byte). List[boolean] uses 1 byte per bool. Prefer Bitvector/Bitlist.
  5. Zero vs empty. is_zero(x) checks if x equals its type's default value. An empty list is zero. A list with one zero element is not.

üß† Summary

SSZ is more than a serialization format‚Äîit's a canonical Merkle structure for every consensus type. This enables:

  1. Deterministic hashing ‚Äî same object, same root, every client
  2. Light client proofs ‚Äî prove any field with ~32 * log‚ÇÇ(n) bytes
  3. Generalized indices ‚Äî address any tree node with one integer
  4. Efficient multiproofs ‚Äî prove k fields cheaper than k separate proofs
