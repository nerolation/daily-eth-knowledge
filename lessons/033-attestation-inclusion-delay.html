📚 Lesson 33: Attestation Inclusion Delay

The timing window that turns raw attestations into on-chain state transitions

March 9, 2026 • Module 4: Attestations Deep Dive

🔍 Spec Location

  • phase0/beacon-chain.md → Time parameters, process_attestation
  • altair/beacon-chain.md → get_attestation_participation_flag_indices

💡 Core Concept

Attestations can't be included in the same slot they attest to. There's a mandatory gap:

# Time parameters from phase0/beacon-chain.md
MIN_ATTESTATION_INCLUSION_DELAY = 1 slot (12 seconds)
Maximum window = SLOTS_PER_EPOCH = 32 slots

This creates a timing constraint that affects both correctness and rewards:

assert data.slot + MIN_ATTESTATION_INCLUSION_DELAY <= state.slot <= data.slot + SLOTS_PER_EPOCH

An attestation for slot N can be included in slots N+1 through N+32 (same epoch or the following epoch boundary).

Why the 1-slot minimum?

When you attest, you're voting for the block at data.slot. That block hasn't been processed yet when you create the attestation—you only see its root. The proposer of slot N+1 needs to:

  1. See your attestation (network propagation)
  2. Verify it against processed state
  3. Include it in their block

Including in the same slot would require attesters to vote on a block they haven't fully validated. The 1-slot delay ensures attesters have seen and processed the block they're voting for.

🛠 Phase 0: The Inclusion Delay Reward

In Phase 0, inclusion delay directly affected rewards through inverse scaling:

class PendingAttestation(Container):
    aggregation_bits: Bitlist[MAX_VALIDATORS_PER_COMMITTEE]
    data: AttestationData
    inclusion_delay: Slot  # state.slot - data.slot
    proposer_index: ValidatorIndex
def get_inclusion_delay_deltas(state, ...):
    for index in attesting_indices:
        attestation = min(attestations, key=lambda a: a.inclusion_delay)
        max_attester_reward = base_reward - proposer_reward
        rewards[index] += max_attester_reward // attestation.inclusion_delay
Inclusion Slot Delay Reward
N+1 1 100% of max
N+2 2 50% of max
N+5 5 20% of max

This created strong economic pressure for fast inclusion, but also some problems:

  • Validators near the slot boundary got disadvantaged
  • Network latency variations caused unfair reward differences
  • The proposer who included an attestation also got rewarded, creating inclusion races

🛠 Altair: Participation Flags with Timeliness

Altair replaced the continuous reward scaling with discrete timeliness thresholds:

def get_attestation_participation_flag_indices(
    state, data, inclusion_delay
) -> Sequence[int]:
    participation_flag_indices = []
    
    # Source: must be included within sqrt(32) ≈ 5 slots
    if is_matching_source and inclusion_delay <= integer_squareroot(SLOTS_PER_EPOCH):
        participation_flag_indices.append(TIMELY_SOURCE_FLAG_INDEX)
    
    # Target: must be included within same epoch (32 slots)
    if is_matching_target and inclusion_delay <= SLOTS_PER_EPOCH:
        participation_flag_indices.append(TIMELY_TARGET_FLAG_INDEX)
    
    # Head: must be included in exactly the next slot
    if is_matching_head and inclusion_delay == MIN_ATTESTATION_INCLUSION_DELAY:
        participation_flag_indices.append(TIMELY_HEAD_FLAG_INDEX)
    
    return participation_flag_indices
Flag Deadline Weight Rationale
TIMELY_HEAD 1 slot 14/64 Fork choice needs fresh votes
TIMELY_SOURCE 5 slots 14/64 FFG source still useful briefly
TIMELY_TARGET 32 slots 26/64 FFG target valid all epoch

❓ Why This Design?

Head votes have the strictest deadline (1 slot) because LMD-GHOST fork choice runs every slot. A head vote for slot N arriving at slot N+3 is useless—fork choice has already moved on. Might as well give no credit.

Source votes get 5 slots because they contribute to justification, which happens at epoch boundaries. But they're still time-sensitive for network consistency. The sqrt(32) choice is somewhat arbitrary but provides a reasonable window.

Target votes get the full epoch because they directly contribute to finalization. A target vote for epoch E is valid anywhere in epoch E or E+1 (until epoch boundary processing). Maximum flexibility here because finality is the priority.

The Phase 0 → Altair shift moved from "continuous penalty for delay" to "binary thresholds." This is fairer: a validator with 500ms network latency isn't perpetually disadvantaged vs one with 50ms. Either you make the deadline or you don't.

⚠️ Gotchas

1. Head rewards are basically MEV-adjacent

To get TIMELY_HEAD credit, your attestation must be in the very next block. If that proposer doesn't include you (or misses their slot), you lose 14/64 of your attestation reward. This creates proposer-attester coordination incentives.

2. The epoch boundary is sharp

An attestation for slot 31 of epoch E can be included in slots 32-63 (epoch E+1). But the check is data.slot + SLOTS_PER_EPOCH, not "same or next epoch." There's a subtle off-by-one trap for implementations.

3. Double-inclusion protection

Altair uses participation flags, not attestation lists. Once epoch_participation[index] has a flag set, including another attestation from that validator for the same epoch just wastes block space—no additional reward.
