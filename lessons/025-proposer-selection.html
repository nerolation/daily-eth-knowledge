📚 Lesson 25: Proposer Selection

How Ethereum picks who proposes each block

Overview

Every slot needs exactly one block proposer. The selection must be:

  • Unpredictable — can't be gamed ahead of time
  • Weighted by stake — more ETH = proportionally more proposals
  • Deterministic — every node computes the same proposer
🔍 Spec Location: phase0/beacon-chain.md → get_beacon_proposer_index, compute_proposer_index

💡 Core Concept

Proposer selection is a weighted random sample from all active validators. The algorithm:

  1. Derive a slot-specific seed from RANDAO
  2. Iterate through shuffled validator indices
  3. Accept or reject each candidate based on effective balance
P(accept) = effective_balance / MAX_EFFECTIVE_BALANCE

A 32 ETH validator has 100% accept chance when selected. A 16 ETH validator (if it existed pre-Electra) would have 50%.

🛠 The Code — get_beacon_proposer_index

def get_beacon_proposer_index(state: BeaconState) -> ValidatorIndex:
    epoch = get_current_epoch(state)
    # Mix slot into seed for per-slot uniqueness
    seed = hash(get_seed(state, epoch, DOMAIN_BEACON_PROPOSER) 
                + uint_to_bytes(state.slot))
    indices = get_active_validator_indices(state, epoch)
    return compute_proposer_index(state, indices, seed)

Key insight: The seed combines:

  • RANDAO mix (from MIN_SEED_LOOKAHEAD epochs ago)
  • DOMAIN_BEACON_PROPOSER (domain separation)
  • Epoch (hashed into the seed)
  • Slot (hashed on top)

This ensures different proposers for different slots, even within the same epoch.

🛠 The Code — compute_proposer_index (Phase 0)

def compute_proposer_index(
    state: BeaconState, 
    indices: Sequence[ValidatorIndex], 
    seed: Bytes32
) -> ValidatorIndex:
    assert len(indices) > 0
    MAX_RANDOM_BYTE = 2**8 - 1  # 255
    i = uint64(0)
    total = uint64(len(indices))
    
    while True:
        # Pick a shuffled candidate
        candidate_index = indices[
            compute_shuffled_index(i % total, total, seed)
        ]
        # Get randomness for this iteration
        random_byte = hash(
            seed + uint_to_bytes(uint64(i // 32))
        )[i % 32]
        
        effective_balance = state.validators[
            candidate_index
        ].effective_balance
        
        # Accept/reject based on stake weight
        if (effective_balance * MAX_RANDOM_BYTE >= 
            MAX_EFFECTIVE_BALANCE * random_byte):
            return candidate_index
        i += 1

🔬 Dissecting the Inner Loop

Step 1 — Candidate Selection

compute_shuffled_index(i % total, ...) wraps around the validator list. First iteration tries index 0's shuffled position, then 1, etc.

Step 2 — Fresh Randomness

Each batch of 32 iterations shares a hash (amortized cost). Within the batch, each iteration uses a different byte.

Step 3 — Weighted Acceptance

effective_balance * 255 >= MAX_EFFECTIVE_BALANCE * random_byte

For 32 ETH validator (max effective):

  • 32e9 * 255 >= 32e9 * random_byte
  • Simplifies to: 255 >= random_byte
  • Always true — 100% acceptance

For hypothetical 16 ETH:

  • 16e9 * 255 >= 32e9 * random_byte
  • Simplifies to: 127.5 >= random_byte
  • ~50% acceptance (random_byte ∈ [0, 255])

⚡ Electra Modification (EIP-7251 MaxEB)

With variable effective balance up to 2048 ETH, the algorithm needed more precision:

def compute_proposer_index(...) -> ValidatorIndex:
    assert len(indices) > 0
    MAX_RANDOM_VALUE = 2**16 - 1  # 65535 (was 255)
    i = uint64(0)
    total = uint64(len(indices))
    
    while True:
        candidate_index = indices[
            compute_shuffled_index(i % total, total, seed)
        ]
        # [Modified] 2 bytes instead of 1
        random_bytes = hash(seed + uint_to_bytes(i // 16))
        offset = i % 16 * 2
        random_value = bytes_to_uint64(
            random_bytes[offset : offset + 2]
        )
        effective_balance = state.validators[
            candidate_index
        ].effective_balance
        
        # [Modified] Compare against 2048 ETH max
        if (effective_balance * MAX_RANDOM_VALUE >= 
            MAX_EFFECTIVE_BALANCE_ELECTRA * random_value):
            return candidate_index
        i += 1

Why the change?

  • 8-bit random gives 256 "buckets" — too coarse for 2048 ETH range
  • 16-bit random gives 65536 buckets — smooth distribution
  • A 2048 ETH validator now has 64× the proposal chance of 32 ETH

❓ Why This Design

Q: Why not just pick proportionally by stake directly?
A: Would require computing cumulative stake sums, binary search. The loop is simpler and terminates quickly in practice.

Q: How many iterations on average?
A: With all validators at MAX_EFFECTIVE_BALANCE, exactly 1 iteration (100% acceptance). With mixed balances, still typically 1-2.

Q: What if infinite loop?
A: Impossible — at least one validator has positive effective balance, and the acceptance condition eventually triggers.

⚠️ Gotchas

1. Proposer is slot-dependent, not block-dependent
Same slot → same proposer, regardless of which parent block you're building on.
2. RANDAO manipulation limited
An adversary controlling the last proposer before seed finalization can bias by 1 bit — but seed is locked MIN_SEED_LOOKAHEAD (1 epoch) before use.
3. The loop looks unbounded but isn't
Maximum iterations proportional to MAX_EFFECTIVE_BALANCE / min(effective_balance). With realistic balances, terminates fast.
4. Slashed validators can still be proposers
get_active_validator_indices includes slashed validators until exit_epoch. They're active, just penalized.
